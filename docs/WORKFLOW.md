# WORKFLOW — как вести RnD в этом репозитории

Этот документ фиксирует рабочий процесс: как устроен RnD, какие ноутбуки в нём живут,
как результат передаётся во **внутренний контур** и что обязательно проверить перед push.
Правила здесь — источник правды по процессу; карта репозитория и канон продублированы в
[`README.md`](../README.md), [`CLAUDE.md`](../CLAUDE.md), [`AGENTS.md`](../AGENTS.md).

## Два контура

- **Внешний контур (этот репозиторий).** Здесь идёт исследование: синтетика с фиксированным
  seed и, если удалось собрать, реальные внешние open-source данные. Можно симулировать
  «несколько датасетов», прогонять Monte-Carlo, сравнивать методы.
- **Внутренний контур.** Сюда ноутбук переносится **вручную, одним файлом** и проверяется,
  как правило, на **одной реальной таблице**. Это не запрещает бутстрап и многократные
  прогоны внутри одной таблицы — если того требует сам RnD.

## Что лежит в папке RnD

```text
rnd/0N_<topic>/
  notebook.ipynb            # обязателен — исследовательская витрина поверх src/rnd_reports/
  notebook_internal.ipynb   # опционален — самодостаточный ноутбук для внутреннего контура
  report.md                 # обязателен — единственный источник содержания отчёта
  report.pdf                # опционален — собирается только для RnD из PDF_ENABLED
  README.md                 # обязателен — короткое описание RnD
```

- Никаких прочих файлов (`*.csv`, `*.png`, `*.py`, `*.txt`) в папке RnD. Единственное
  исключение — `notebook_internal.ipynb`. Артефакты идут в `results/0N_<topic>/`,
  переиспользуемый код — в `src/rnd_reports/`, контекст — в `docs/`.
- Директории `exports/` **нет**: передаваемый ноутбук живёт внутри своей папки RnD.

## `notebook.ipynb` — исследовательский ноутбук

- Тонкая витрина: алгоритмическая логика в `src/rnd_reports/`, ноутбук импортирует её
  (`from rnd_reports.<pkg> import ...`) и показывает результаты.
- Данные: синтетика с фиксированным seed и/или внешние open-source датасеты через
  адаптеры/loaders из gitignored `data/0N_<topic>/`. Можно несколько датасетов/сценариев.
- Все комментарии и markdown — **на русском**.

## `notebook_internal.ipynb` — ноутбук для внутреннего контура

Заводится для RnD, который имеет смысл проверять на **одной таблице** во внутреннем контуре
(табличные RnD). Требования:

1. **Самодостаточность.** Весь нужный код пакета **вшит в ячейки** (`# === Вшито из
   rnd_reports.<pkg>/<mod>.py ===`). Импортов `rnd_reports` нет — ноутбук запускается одним
   файлом там, где есть только базовый стек (`numpy / pandas / scipy / statsmodels / ...`).
2. **Чтение из CSV-переменной.** Есть ячейка-конфиг с `INPUT_CSV = "..."` и описанием
   контракта таблицы. Во внутреннем контуре переключается на реальную таблицу — это
   единственная правка, которую делает получатель.
3. **Ячейка-генератор синтетики.** Отдельная, явно помеченная «⚠️ НЕ запускать во внутреннем
   контуре» ячейка синтезирует **одну** таблицу и пишет её в `INPUT_CSV` (каталог
   `data/0N_<topic>/`, gitignored). Во внутреннем контуре её пропускают (там реальные данные);
   в репозитории она запускается, чтобы проверить запускаемость ноутбука целиком.
4. **Загрузка и анализ.** `df = pd.read_csv(INPUT_CSV)`, далее разбор на одной таблице; где
   уместно — бутстрап как опция.
5. Все комментарии — **на русском**.

Структура ячеек по порядку: шапка (markdown) → импорты → вшитый код → конфиг (`INPUT_CSV`) →
ячейка-генератор → загрузка → анализ → сводка.

## Глобальные правила

- **Перед push: все изменённые ноутбуки перезапускаются до конца и проходят без ошибок.**
  `notebook_internal.ipynb` прогоняется тут же, в репозитории (его захватывает автодискавери).
- Ноутбуки коммитятся исполненными, с актуальными output.
- Комментарии и отчёты — на русском.
- Реальные/корпоративные/скачанные данные **никогда не коммитятся** (только gitignored
  `data/0N_<topic>/`). Сгенерированный `sample.csv` тоже не коммитим — он регенерируемый.
- PDF опционален: см. `PDF_ENABLED` в `tools/generate_pdf.py`.
- Закрытые/приостановленные RnD (01–05) не трогаем без явного запроса.

## Команды

```bash
.venv/bin/python tools/execute_notebooks.py     # выполнить ВСЕ rnd/*/notebook*.ipynb
.venv/bin/python tools/execute_notebooks.py rnd/08_multiple_testing/notebook_internal.ipynb  # точечно
.venv/bin/python tools/generate_pdf.py          # PDF только для RnD из PDF_ENABLED
.venv/bin/python tools/generate_pdf.py rnd/09_topic/report.md  # разовая сборка PDF нового RnD
.venv/bin/python -m pytest -q                   # тесты пакета
```

## Чек-лист перед push

1. Изменённые `notebook.ipynb` и `notebook_internal.ipynb` перезапущены до конца, без ошибок.
2. `pytest` зелёный.
3. PDF (если нужен) пересобран; новый RnD добавлен в `PDF_ENABLED` либо собран явным аргументом.
4. `git status`: нет случайно закоммиченных данных (`data/`, `sample.csv`).
5. Комментарии в ноутбуках — на русском.
